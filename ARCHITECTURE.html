System Architecture

Data Flow — filter change to render

How a user selecting a warehouse in the global filter bar propagates through the stack. FilterBar never emits events — it mutates singleton refs directly, and each view independently watches them.

1
User Input
User picks a warehouse in <select v-model="selectedLocation">. components/FilterBar.vue
2
Shared State
v-model writes to the module-level ref — declared outside the composable, so every consumer sees the same instance. composables/useFilters.js — selectedLocation = ref('all')
3
Reactive Watch
Active view's watch([selectedPeriod, selectedLocation, …], loadData) fires. views/Dashboard.vue · Orders.vue · Inventory.vue
4
Build Params
getCurrentFilters() maps UI refs → API param names: { warehouse, category, status, month }. composables/useFilters.js
5
HTTP Request
api.js builds URLSearchParams, skipping any value equal to 'all', and calls axios. GET /api/dashboard/summary?warehouse=Tokyo
6
Route Handler
FastAPI endpoint receives typed Optional[str] query params. server/main.py — get_dashboard_summary(warehouse, category, status, month)
7
In-Memory Filter
apply_filters() and filter_by_month() list-comprehend over module-level lists; 'all' is a no-op. Quarters resolved via QUARTER_MAP. server/main.py
8
Serialize
Sums/counts computed, validated against Pydantic response_model, returned as JSON.
9
Render
View assigns response to a ref → dependent computed properties recalculate → template re-renders. summary.value = await api.getDashboardSummary(filters)
